SCORM 1.2

SCORM Compliance

How and why this LMS meets the Sharable Content Object Reference Model standard.

Overview
What is SCORM?

SCORM (Sharable Content Object Reference Model) is a set of technical standards developed by the ADL Initiative that governs how e-learning content and Learning Management Systems communicate. It defines three things:

  1. How content is packaged — so any SCORM-compliant LMS can import it
  2. How content is launched — within an LMS environment
  3. How content and the LMS talk at runtime — tracking progress, scores, time, and completion

This LMS implements SCORM 1.2, the most widely adopted version, supported by virtually all authoring tools and learning platforms worldwide.

Benefits
Why SCORM Compliance Matters
  • Interoperability — Content authored in Articulate Storyline, Adobe Captivate, Adapt, H5P, or any SCORM-compliant tool can be imported and delivered through this LMS
  • Portability — Our course content can be exported as a SCORM package and delivered on Moodle, Blackboard, Canvas, SCORM Cloud, or any other compliant platform
  • Standardised tracking — Learner progress, scores, completion status, session time, and bookmarking all follow the same specification regardless of the authoring tool
  • Institutional acceptance — Educational institutions and corporate training departments require SCORM support as a baseline for any LMS
  • Future-proof — SCORM 1.2 has been the stable standard since 2001 with universal support, providing a solid foundation to later add xAPI or cmi5
Architecture
Three Pillars of SCORM Compliance
1. Content Aggregation Model

Defines how content is structured and packaged. Our LMS includes a valid imsmanifest.xml that declares all 12 worksheets as Sharable Content Objects (SCOs) within a hierarchical organisation (Bronze / Silver / Gold tiers).

2. Runtime Environment

Defines how SCOs communicate with the LMS during a session. Our LMS exposes the window.API JavaScript object with all 8 required methods. SCOs find this via the standard window hierarchy search.

3. Sequencing & Navigation

Defines how learners move between SCOs. Our LMS supports free navigation (learners can jump to any worksheet) with prerequisite awareness via the Bronze → Silver → Gold tier structure.

Packaging
Content Aggregation Model

The course is packaged with a valid imsmanifest.xml that declares:

  • 12 SCOs — each worksheet is a Sharable Content Object with its own tracking lifecycle
  • 3-tier organisation — Bronze (WS 1–7), Silver (WS 8–10), Gold (WS 11–12)
  • Mastery score — 40% minimum across all SCOs (Bronze threshold)
  • Resource declarations — every HTML file, image, and script file is listed with correct types
  • Asset resources — shared media (logos, cover images) declared as non-trackable assets
Manifest location: imsmanifest.xml in the package root, following the IMS Content Packaging 1.1.2 specification with ADL SCORM 1.2 extensions.
Session Lifecycle
How Each Worksheet Communicates

When a learner opens a worksheet, the following SCORM session lifecycle executes:

  1. LMSInitialize("") — Session begins, previous state is restored (bookmark, suspend data, scores)
  2. LMSGetValue("cmi.core.entry") — Check if this is a new attempt or a resume
  3. Learner interacts — Completing challenges triggers LMSSetValue for scores, interactions, and status
  4. LMSSetValue("cmi.core.lesson_status", ...) — Status updates: "incomplete" while working, "completed" when all challenges are done
  5. LMSSetValue("cmi.interactions.n.*", ...) — Each challenge is logged as a SCORM interaction with ID, type, and result
  6. LMSCommit("") — Data persisted to storage after each change
  7. LMSFinish("") — Session time calculated and accumulated into total_time
